md-sdlc-wf

Turn a Confluence PRD into a grounded implementation handoff. Parses the spec, reads Figma, loads repo-aware context, hands off to your project's implementation agent AC-by-AC. Closes the loop with a local AC coverage check before PR.

/plugin marketplace add github.com/MD-Sean/WTF

What it is

A Claude Code plugin that adds two layers to your existing SDLC defect filter — one upstream of code, one downstream of code. It does not replace what you already have. Rules, hooks, lint, type-check, code review — all unchanged. The plugin chips in where intent is captured and where coverage is verified.

L0Intent capture

PRD parsed, Figma read, repo context loaded — before a line of code.

L1-4Existing filter

Rules · hooks · lint · code review. Untouched. Your discipline, your patterns.

L5AC coverage

Local pre-PR check that tests actually exercise each acceptance criterion.

L6You ship

Same merge process. Same review path. Less guesswork before, fewer gaps at the end.

What it deliberately does not do

v0.1.0 stays out of: Jira ticket creation, PR title enforcement, CI coverage gates, test scaffolding (TDD lives in your implementation agent), feature-flag wiring suggestions, analytics story proposal, cross-repo coordination, cost telemetry. These were considered and dropped for v1 — the goal is a slim chip-in that ships fast and earns trust before adding scope.

Before / After

What changes when an engineer picks up a new spec.

Day-one PRD triage

Before — manual triage
  • Open Confluence, read 12-page PRD top to bottom
  • Mentally extract Acceptance Criteria from the table
  • Click through to the Figma file, find the right cover frame
  • Skim related-docs links, hope nothing critical is buried
  • Type up a personal summary in a scratch doc
~30–60 min
After — /digest-spec
  • Paste Confluence URL into /digest-spec
  • Dense ≤100-line render: status, AC table, scope, links, rollback
  • Loud warnings if status is Draft, AC missing, or Figma absent
  • Same digest is cached, reused by every other skill
~30 sec

Starting implementation on a story

Before — cold start
  • grep the repo for component names mentioned in AC
  • Read .claude/rules/ manually, hope to remember conventions
  • Hunt for similar prior PRs via git log
  • Find a test file with the right style to mimic
  • Cross-reference Figma tokens by eye
  • Start coding, discover wrong folder structure 20 min in
~45–90 min per AC group
After — /pilot
  • Run /pilot once on the PRD URL
  • Get a context pack per AC group: rules summary, ≤15 ranked files, ≤5 prior PRs, ≤3 test patterns
  • Figma tokens + Code Connect paths pre-mapped
  • Implementation agent receives the handoff pre-grounded
~3–5 min (mostly parallel)

Pre-PR coverage check

Before — hope
  • Push branch, raise PR
  • Reviewer (or code-reviewer agent) spots an AC with no test
  • Send back for rework
  • Re-push, re-review cycle
~1 review-cycle delay
After — /qa-check
  • Run /qa-check on the branch locally
  • Per-AC verdict: covered / partial / uncovered
  • Gaps surfaced with the AC's "Then" clause as a suggested test case
  • Orphan tests flagged
  • Fix before reviewer ever sees the gap
~10 sec per check

The 5-phase pipeline

/pilot <PRD-url-or-Jira-epic> runs the full flow. Phases 1–2 in parallel. The skill halts after every phase so a human picks continue / edit / abort.

   │

   │

Where the new and existing pieces meet

The pipeline is deliberately bookended around the existing .claude/ defect filter. New components only at the upstream end (Phase 1–5) and the downstream end (/qa-check). Everything between — senior-frontend-engineer writing code under hooks + rules + TDD, then senior-code-reviewer auditing for quality — runs exactly as it does today, unchanged.

StageSourceConcern
Spec digest new Parse PRD layout-agnostic → JSON. Status gate. Drift detection via Confluence version id.
Figma read new Cover frame, tokens, Code Connect mapping. Graceful degrade if Figma MCP absent.
Context loading new Distill .claude/rules/ + repo files + prior PRs + test patterns into a per-AC-group handoff.
Implementation existing senior-frontend-engineer agent. Writes code + tests under TDD discipline.
Hook enforcement existing Pre-edit + post-edit shell hooks: mantine-styling, lint, type-check, auto-format, protect-files, block-dangerous.
Code review existing senior-code-reviewer agent. Quality, perf, security, rule adherence.
AC coverage check new qa-validator. Static AC ↔ test mapping with PASS/WARN/FAIL verdict.

Resumable state

Every run writes scratch to .pilot/<spec-id>/ in the current repo:

.pilot/
└── <spec-id>/
    ├── digest.json           ← spec-digester output (with confluence_version_id)
    ├── figma.json            ← figma-reader output
    ├── adr.md                ← optional, if "Technical Approach" section present
    ├── context/
    │   ├── G1.md             ← per-AC-group context pack
    │   └── ...
    └── .last-accessed        ← TTL marker (90-day GC after Done)

On rerun, spec-digester checks the Confluence page's version id against the cached digest. If unchanged, cache is reused — no re-fetch. If drifted, the agent auto-refreshes and emits a spec_drifted warning.

How it chips into existing .claude/ systems

Repos that already use Claude Code with rules, hooks, agents, and skills don't need to change anything. The plugin adds agents alongside existing agents, skills alongside existing skills. context-loader reads whatever .claude/rules/ the target repo has and surfaces them to the implementation agent. Your conventions stay yours.

/qa-check — local AC coverage gate

Run before raising a PR. Resolves the linked spec, finds tests claiming AC coverage, statically verifies they exercise the Given/When/Then. Outputs PASS / WARN / FAIL. Advisory — does not block merges.

The 6 agents

Each agent runs in an isolated context window. Heavy reads (PRD text, Figma screenshots, dozens of repo files) stay out of the main conversation. Each agent is callable solo by name when you don't need the whole pipeline.

Plugin ships 4 new agents for the upstream + downstream chip-in, plus 2 bundled agents that the existing 4-layer defect filter relies on (so adopters get a complete pipeline out of the box, regardless of whether their repo already has these).

New — added by the plugin

AgentReads fromReturns
spec-digester Atlassian MCP (Confluence + Jira) Canonical JSON with AC matrix, scope, metrics, Figma refs, related docs, status, confluence_version_id for drift detection. Layout-agnostic.
figma-reader Figma MCP (read-only) Cover-frame screenshot path, design tokens used, component instances with Code Connect mapping, sub-frame inventory with node IDs.
context-loader Repo files, .claude/rules/, GitHub MCP / git log Markdown context pack: rules to honor, ranked relevant files (≤15), prior related PRs (≤5), test patterns to mimic (≤3).
qa-validator Spec digest, branch git diff, test files Per-AC coverage report with covered / partial / uncovered verdict, plus orphan test detection.

Bundled — for the existing 4-layer filter

AgentRole in pipelineNotes
senior-frontend-engineer Implementation Receives AC-group handoff from /pilot. Writes code under the project's rules + hooks + TDD discipline. Stack-aware (Next.js 16, React 19, TypeScript strict, Mantine 8, Tailwind 4). Adopters on a different stack should fork this agent prompt to match their stack.
senior-code-reviewer Quality review Audits engineer output for race conditions, stale closures, SSR correctness, missing cleanup, project rule adherence, perf, security. Not AC coverage — that's qa-validator's concern.

Bundled agents come pre-configured for the Mudah web stack. If your repo already has agents of these names, the plugin's versions are loaded by Claude Code's standard precedence (project .claude/ beats plugin). Repo-local customizations always win.

Why agents, not skills

Skills load into the main conversation context. Agents have their own isolated context window. PRD text, Figma frame metadata, and dozens of repo file paths would each balloon the main context — so they live in agents that return only summaries. The 3 skills are slash-command wrappers with policy and rendering; the 6 agents do the actual heavy work.

Solo invocation

When you don't need the full pipeline, invoke an agent by name:

"use spec-digester to pull just the rollback triggers from page 4862836741"
"use figma-reader on frame 867:6347 and tell me which components lack Code Connect"
"use context-loader to find relevant files for an API endpoint returning 20 active cars"
"use qa-validator on the current branch deeply, even for partial coverage"

Common objections

ObjectionHonest response
"I already read PRDs fast. I don't need a digest."The digest isn't only for you. It's for the engineer joining the squad next month. Cost is 30 seconds. Run it once; if it tells you nothing new, fine.
"I don't trust an AI-generated summary."Don't trust it blindly. AC table is preserved verbatim — IDs, scenarios, Given/When/Then. Compression is only around the prose. Verify by spot-checking AC.
"It'll break when MCP is down."/digest-spec returns cached digest with a warning. figma-reader degrades gracefully. Engineer is never blocked outright.
"We already have a senior-code-reviewer."Different concerns. Reviewer judges code quality. /qa-check judges AC coverage. Both are bundled. Use both.
"My team isn't on Claude Code."Plugin is engineer-side. PMs, designers, QA don't change anything. Confluence + Figma + Jira workflow stays the same.
"My PRDs don't have AC matrices."Digest warns loudly with no_acceptance_criteria. That's the plugin doing its job. Push back to the PM.
"Will .pilot/ bloat my repo?"/pilot auto-adds .pilot/ to .gitignore. Local scratch only. 90-day TTL after spec status = Done.

How to know it's working

After 2 months of squad-level adoption, look for these leading indicators:

  • Engineer-side: PR comments reference SPEC-XXX AC-07 instead of "what does this AC mean"
  • Reviewer-side: code-review feedback shifts toward quality, not coverage
  • PM-side: PMs start fixing their own PRDs based on /digest-spec warnings
  • Standup-side: people reference spec IDs + AC group IDs (G3) instead of explaining specs verbally

Don't gate adoption decisions on trailing metrics (defect rates, time-to-merge) — those shift in months. Gate on the leading ones above.

When to remove it

Honest exit ramp. Plugin failed if:

  • Engineers consistently bypass /digest-spec after 4-week honest use
  • /qa-check produces too many false positives (engineers learn to ignore)
  • MCP outages exceed 10% of runs and persist >2 weeks
  • Adoption stalls at L1 for 3+ months despite organic growth attempts

Uninstall is clean: /plugin uninstall md-sdlc-wf then /plugin marketplace remove …. No data migration. No vendor lock-in. Restart any time.
